Wire the Contact form end-to-end (frontend + backend) so submissions actually send and are recorded.

Scope/Requirements

1) Frontend (React + React Hook Form + Zod)

- On the Contact page, implement a form with fields:
    - name (required, 2–100 chars)
    - email (required, valid email)
    - subject (required, 3–120 chars)
    - message (required, 10–3000 chars)
- Validate with Zod and show inline errors.
- Submit to `POST /api/contact` with JSON body.
- UX:
    - Disable submit + show loading state while sending
    - Show clear success message on 200
    - Show clear error message on 4xx/5xx (do not leak stack traces)

2) Backend (Express v5, port 5000)

- Add route: `POST /api/contact`
- Validate request body with Zod on the server as well.
- Add basic abuse protection:
    - simple rate limit per IP (e.g., 5/min) OR lightweight in-memory throttling
    - honeypot field support (frontend includes hidden “company” field; if filled, silently return 200 but do nothing)

3) Delivery method (pick one and implement fully)

Option A (preferred if already in stack): Mailgun / Nodemailer

- Send an email to the farm admin inbox (env var `CONTACT_TO_EMAIL`)
- Set Reply-To to the user’s email
- Subject format: `[Contact] {subject}`
- Include name/email/message in the body

Option B: Zapier webhook (if that’s what you’re using elsewhere)

- POST payload to `process.env.ZAPIER_CONTACT_WEBHOOK_URL`
- Handle non-2xx from Zapier as an error

4) Optional: persistence (recommended)

- Add `contact_submissions` table (Drizzle + Postgres) with:
    - id, name, email, subject, message, created_at, ip (optional), user_agent (optional), status (“sent”/“failed”)
- Insert a row on submission; update status after sending succeeds/fails.

5) Secrets / config (no hardcoding)

- Read provider keys and recipient email from `process.env` (Replit Secrets):
    - `CONTACT_TO_EMAIL`
    - plus Mailgun or SMTP creds OR Zapier webhook URL

Definition of Done

- Submitting the contact form results in a real delivered message to `CONTACT_TO_EMAIL` (or a Zapier receipt), and the UI shows success.
- Invalid inputs are blocked client-side and server-side.
- TypeScript builds cleanly for both frontend and backend.
- No credentials are hardcoded; everything uses environment variables.
