Implement CMS Pages + Menus + strong SEO, served at `/p/:slug`, with root aliases for `/about`, `/faq`, `/contact` mapping to CMS slugs. Update routing to match the existing Wouter setup shown below (no root catch-all).

A) Add CMS page renderer + routes (frontend)

1) New public page component

- Create `src/pages/CmsPage.tsx`
- It should accept a `slug` and fetch the published CMS page via `GET /api/cms/pages/:slug`
- Render:
    - page title
    - `content_html` (sanitized render)
- Apply SEO tags using the returned SEO fields (meta title/description/canonical/OG/robots).

2) Add `/p/:slug` route (new)

In `App.tsx`, add before the catch-all:

- `Route path="/p/:slug"` that wraps `CmsPage` in `PublicLayout`.

3) Replace existing root pages with CMS aliases (keep the URL, swap implementation)

In `App.tsx`, for these routes:

- `/about`
- `/faq`
- `/contact`

Replace the current components (`About`, `Faq`, `Contact`) with the CMS renderer using slugs:

- `/about` → slug `about`
- `/faq` → slug `faq`
- `/contact` → slug `contact`

Important: do NOT add a generic root `/:slug` CMS catch-all (your app already has lots of root routes; aliases should be explicit only).

B) CMS data model (backend + DB)

1) Drizzle tables

Create:

- `cms_pages`: id (serial), slug (unique), title, content_html, status (draft/published), published_at, created_at, updated_at
- `cms_page_seo`: page_id (unique FK), meta_title, meta_description, canonical_url, og_title, og_description, og_image_url, robots (index_follow/noindex_nofollow), created_at, updated_at

Slug validation (shared FE/BE)

- slug regex: `^[a-z0-9]+(?:-[a-z0-9]+)*$`
- reserved slugs must be blocked (because they’re real routes in this app):
    - `admin`, `api`, `auth`, `account`, `shop`, `cart`, `checkout`, `order-confirmation`, `pickup-events`, `unsubscribe`, `policies`, `how-we-raise-them`, `p`
    - plus anything else already defined in App.tsx routes

2) API routes (Express + Zod)

Admin-only (requireAdmin):

- `GET /admin/cms/pages`
- `POST /admin/cms/pages`
- `GET /admin/cms/pages/:id`
- `PATCH /admin/cms/pages/:id`
- `POST /admin/cms/pages/:id/publish`
- `POST /admin/cms/pages/:id/unpublish`
- `GET /admin/cms/pages/:id/seo`
- `PATCH /admin/cms/pages/:id/seo`

Public:

- `GET /api/cms/pages/:slug` (published only; 404 if not found/unpublished)

C) Admin UI (pages + menus)

1) Pages admin

- Add routes + UI under Admin:
    - `/admin/pages` list
    - `/admin/pages/new` create
    - `/admin/pages/:id` edit (Tiptap WYSIWYG) + publish/unpublish + SEO panel
- Use RHF + Zod + Shadcn components.

2) Menus admin

- Add `cms_menus` and `cms_menu_items` tables
- Add `/admin/menus` UI to edit Header/Footer menus (add/edit/reorder/hide)
- PublicLayout must render header/footer nav from DB (no hardcoded nav), with safe fallback.

D) Strong SEO (per-page + site-wide)

1) Per-page SEO (editable)

- Use cms SEO fields to set `<title>`, meta description, canonical, OG/Twitter, robots.

2) Site-wide

- Implement:
    - `/robots.txt` (points to sitemap)
    - `/sitemap.xml` includes:
        - core routes you want indexed: `/`, `/shop`, `/how-we-raise-them`, `/pickup-events`, `/policies/sales-returns`
        - `/p/:slug` for all published CMS pages with robots=index

        - product detail pages (your current product route is `/shop/:id`; include those)

3) Make SEO “real” (not SPA-only)

If SSR is too big, implement prerendering at deploy/build time for published CMS pages and serve static HTML with hydration.
