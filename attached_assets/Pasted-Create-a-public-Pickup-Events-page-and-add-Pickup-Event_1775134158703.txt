Create a public Pickup Events page and add Pickup Event selection at checkout, both powered by the same Admin-created Pickup Events data (single source of truth).

A) Public Pickup Events page (read-only)

1) Public API endpoint

- Add `GET /api/pickup-events` (public, no auth).
- Return only events that are publicly visible:
    - `published = true` (or `is_public = true`)
    - `cancelled = false`
    - `startAt >= now()`
- Sort by `startAt ASC`.
- Return fields (minimum): `id`, `title`, `startAt`, `endAt`, `locationName` (optional), `address` or `instructions`, `notes` (optional).

2) Public page route + UI

- Add route `/pickup-events` (or `/events`).
- Render “Upcoming pickup events” list/cards using `GET /api/pickup-events`.
- Display times in America/Detroit.
- Empty state: “No pickup events currently scheduled. Check back soon.”
- Mobile-friendly styling; match existing site components.

3) Admin publishing control

- Ensure Admin “Pickup Events” supports Published vs Draft (or `is_public` toggle).
- Only Published events appear publicly.

Definition of done (public page)

- Admin creates event → marks Published → it appears on `/pickup-events` immediately.
- Draft/cancelled events do not appear.

B) Checkout: require pickup event selection + write to admin event log

1) Checkout UI (required)

- At checkout, add a required selector (dropdown/radio list) labeled “Select a pickup event”.
- Options come from the same `GET /api/pickup-events` endpoint (published + upcoming only).
- Customer must select exactly one event to place the order.
- If no events available, block checkout with a clear message (fail-closed).

2) Persist selection on the Order

- Add `orders.pickup_event_id` (FK) or equivalent.
- Ensure the selected event is visible on:
    - Order confirmation page
    - Admin order detail page
    - Receipt email (if implemented)

3) Create an attendee/event log row in Admin

- Create/ensure a join table (single source of truth), e.g. `pickup_event_orders` (or `pickup_event_attendees`) with:
    - `pickup_event_id`
    - `order_id`
    - customer name/email/phone snapshot from checkout
    - `created_at`
    - optional `status` (active/cancelled)
- Enforce uniqueness: unique(`pickup_event_id`, `order_id`) so duplicates cannot be created.

4) Admin Pickup Events page: show attendees/orders

- In Admin “Pickup Events” detail view, show a list of orders/customers associated with that event (from the join table).
- If an order is cancelled/refunded, either:
    - mark the attendee row cancelled, or
    - remove it (pick one behavior and implement consistently; recommendation: mark cancelled for audit trail).

5) Integrity rules

- Only allow selecting events that are published, not cancelled, and upcoming.
- If capacity exists and is full, do not show it as selectable (or show disabled with “Full”).

Acceptance checks

- Customer cannot complete checkout without selecting an event.
- After placing an order, the order record has `pickup_event_id` and the customer appears in the Admin event’s attendee/order list.
- Public `/pickup-events` page and checkout selector show the same set of published upcoming events (single source of truth).
